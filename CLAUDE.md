# Personal website — CLAUDE.md

Static personal site and blog built with [Eleventy](https://www.11ty.dev/) (v3). Posts are written in org-mode. The site language is **Spanish**.

## Commands

```bash
npm start       # dev server with live reload at localhost:8080
npm run build   # production build → _site/
```

## Project structure

```
.eleventy.js          # Eleventy config: plugins, filters, input/output dirs
src/
  _data/site.js       # global metadata: title, url, author, language
  _includes/
    base.njk          # base HTML layout (nav, footer, dark-mode toggle)
    home.njk          # wraps base.njk for the homepage
    post.njk          # unused by the org plugin; kept for reference
  assets/style.css    # all styles — no frameworks, no build step
  index.njk           # homepage (shows 5 most recent posts)
  posts/              # org-mode source files (one file = one post)
  writing/
    index.njk         # post listing with tag filter nav
    post.njk          # generates one page per org post via pagination
    etiquetas.njk     # generates one page per tag via pagination
    feed.njk          # Atom feed → /writing/feed.xml
.github/workflows/
  deploy.yml          # builds and deploys to GitHub Pages on push to main
```

## Writing a post

Create a `.org` file in `src/posts/`. Metadata comes from org keywords at the top:

```org
#+TITLE: El título de la entrada
#+DATE: 2026-05-01
#+FILETAGS: :etiqueta-uno:etiqueta-dos:

Contenido aquí.
```

- `#+FILETAGS:` uses the `:tag:` colon-delimited format.
- The URL of the post is derived from the filename: `mi-entrada.org` → `/writing/mi-entrada/`.
- Posts are sorted newest-first automatically.

## How the org plugin works

`eleventy-plugin-org` is **not** a standard Eleventy template engine. It reads all `.org` files from `src/posts/` independently (configured via `orgDir` in `.eleventy.js`), converts them to HTML, and exposes two collections:

- `collections.org` — all posts, each with `{ data.slug, title, date, tags, content }`
- `collections.orgTags` — array of `{ tagName, posts[] }`, one entry per tag

Post pages are generated by `src/writing/post.njk` using Eleventy pagination over `collections.org` (one page per item). Tag pages work the same way via `etiquetas.njk` over `collections.orgTags`.

**Important:** post metadata lives at `post.data.slug`, `post.title`, `post.date`, `post.tags`, `post.content` — not as standard Eleventy front matter.

## Custom filters (`.eleventy.js`)

| Filter | Usage | Purpose |
|---|---|---|
| `head` | `collection \| head(n)` | First N items of an array |
| `isoDate` | `date \| isoDate` | Formats a date as `YYYY-MM-DD` |
| `stripLeadingSlash` | `post.data.slug \| stripLeadingSlash` | Removes the leading `/` from slugs for building URLs |

The RSS plugin (`@11ty/eleventy-plugin-rss`) also adds `dateToRfc3339` and `getNewestCollectionItemDate`.

## Styles and dark mode

All styles are in `src/assets/style.css`. Design tokens are CSS custom properties on `:root` (light) and `[data-theme="dark"]`. No JavaScript build step — the CSS is copied as-is.

The dark-mode toggle button in `base.njk` sets `data-theme` on `<html>` and persists the choice in `localStorage`. A small inline `<script>` in `<head>` applies the saved theme before paint to avoid flash.

## Deployment

Push to `main` → GitHub Actions builds the site and deploys `_site/` to GitHub Pages via the official Pages actions. Requires **Settings → Pages → Source: GitHub Actions** to be enabled in the repo.

Update `url` in `src/_data/site.js` to match the actual GitHub Pages URL before first deploy.
